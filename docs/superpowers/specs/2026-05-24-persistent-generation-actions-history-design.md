# Устойчивые действия генерации и единая история CardBot

## Цель

Сделать действия `Сохранить как шаблон` и `Изменить и повторить` независимыми от памяти процесса Telegram-бота, а историю генераций - единым списком для режимов `text_only`, `text_and_images` и `images_only`.

После изменения пользователь может сохранить или повторить ранее выполненную генерацию после переходов по меню, рестарта бота или деплоя, если запись была создана в новой схеме.

## Текущая проблема

- Данные для сохранения и повтора лежат только в `context.user_data["last_generation"]`.
- Inline-кнопки результата используют глобальные callback-значения `action:save_template` и `action:repeat_edit`, не указывая запись генерации.
- После рестарта процесса Telegram-сообщение с кнопками остается, но `last_generation` теряется.
- Таблица `generations` хранит текстовый результат, но не хранит `marketplace`, `mode`, фото и настройки изображений.
- Таблица `image_sessions` хранит техническую информацию image-пайплайна, но не является пользовательской историей и не покрывает text-only.

## Решение

### Единый реестр пользовательских запусков

Таблица `generations` становится источником данных для истории, сохранения шаблона и повтора генерации во всех трех режимах.

В `generations` добавляются nullable-поля:

- `marketplace TEXT`;
- `mode TEXT`;
- `photo_file_ids TEXT`;
- `images_count INT`;
- `image_guidance TEXT`.

Существующие output-поля остаются без изменений. Для режима `images_only` они сохраняются пустыми (`NULL`), поскольку текстовая карточка не создается.

`image_sessions` сохраняется как технический журнал выполнения image-пайплайна: промпты, отчет, результаты и стоимость. Он не заменяет пользовательскую историю.

### Создание записей

- `text_only`: при успешной генерации сохраняется запись `generations` с исходным запросом, `marketplace`, `mode="text_only"` и текстовым результатом.
- `text_and_images`: при успешном сохранении текстовой карточки создается одна запись `generations` с исходным запросом, `marketplace`, `mode="text_and_images"`, фото, запрошенным количеством изображений, image guidance и текстовым результатом. Если последующий image-этап завершается ошибкой, уже выданная и списанная текстовая карточка остается корректной записью истории с исходными параметрами запуска. `image_sessions` параллельно хранит технические данные image-этапа.
- `images_only`: после успешной отправки хотя бы одного изображения создается запись `generations` с исходным запросом, `marketplace`, `mode="images_only"`, фото, числом запрошенных изображений и image guidance; output-поля остаются пустыми.

Запись истории должна соответствовать успешному пользовательскому результату или успешной части комбинированного запуска, за которую уже произошло списание. Полностью неуспешные генерации не должны появляться как повторяемые записи.

### Адресные кнопки результата

После новой успешной генерации кнопки результата получают идентификатор строки истории:

- `generation_save:<generation_id>`;
- `generation_repeat:<generation_id>`.

Перед выполнением любого действия бот загружает генерацию из БД по паре `generation_id + user_id`. Запись другого пользователя недоступна.

Старые callbacks `action:save_template` и `action:repeat_edit` остаются совместимым fallback для сообщений, отправленных ранее: они продолжают использовать `last_generation`, если это состояние еще есть в памяти процесса, и возвращают текущее предупреждение, если состояния уже нет.

### Сохранение как шаблон

Сохранение по `generation_save:<id>` использует только повторяемый вход генерации:

- маркетплейс;
- режим;
- исходное описание товара (`input_text`);
- референсные фото, если они были;
- число изображений, если оно было выбрано;
- image guidance, включая уже объединенные пожелания и выбранный стиль.

Ответ бота (`output_title`, `output_description`, `output_keywords`, `output_characteristics`) в шаблон не переносится.

После нажатия кнопки бот запрашивает имя шаблона и хранит выбранный `generation_id` до ввода имени. На шаге сохранения запись повторно загружается из БД и повторно проверяется по владельцу.

### Изменить и повторить

Действие `generation_repeat:<id>` загружает повторяемый вход из БД и запускает существующий flow редактирования:

- для `text_only` - пользователь вводит изменения к исходному описанию;
- для `text_and_images` и `images_only` - сохраняются фото, количество изображений и guidance, далее применяется текущий выбор повторения с теми же или новыми фото.

Повтор не зависит от `context.user_data["last_generation"]`; временное состояние используется только после безопасной загрузки записи из БД в рамках активного диалога.

### История

Команда и кнопка `История` выводят последние пять строк `generations` для текущего пользователя независимо от режима.

Каждая новая запись с полным контекстом показывает кнопку:

- `💾 Сохранить как шаблон` -> `generation_save:<id>`.

Формат истории:

- `text_only`: исходный запрос и текущие текстовые output-блоки;
- `text_and_images`: исходный запрос, текстовые output-блоки и отметка о режиме с изображениями;
- `images_only`: исходный запрос и отметка о режиме/запрошенном количестве изображений без искусственных пустых текстовых output-блоков.

### Старые записи

Существующим строкам `generations` нельзя достоверно восстановить маркетплейс и режим, поскольку эти значения ранее не сохранялись.

Правило совместимости:

- старые строки продолжают отображаться в истории;
- если у строки нет `marketplace` или `mode`, кнопка сохранения не показывается;
- бот не подставляет предполагаемые значения и не создает потенциально неверные шаблоны.

## Изменения компонентов

### `db.py`

- Добавить миграционные `ALTER TABLE generations ADD COLUMN IF NOT EXISTS ...`.
- Расширить сохранение успешной текстовой генерации и возвращать `generation_id`.
- Добавить сохранение успешной image-only записи.
- Расширить `get_recent_generations()` необходимыми полями, включая `id`.
- Добавить owner-safe метод получения повторяемой генерации по `id` и `user_id`.

### `bot_keyboards.py`

- Передавать `generation_id` в клавиатуры после генерации.
- Добавить клавиатуру или builder для кнопки сохранения под строкой истории.
- Сохранить legacy-кнопки для ранее отправленных сообщений.

### `bot.py`

- Получать `generation_id` после сохранения успешного запуска и формировать адресные кнопки.
- Сохранять image-only запуск в единый реестр.
- Обрабатывать `generation_save:<id>` и `generation_repeat:<id>` через owner-safe загрузку из БД.
- Хранить временный id выбранной генерации на этапе ввода имени шаблона.
- Обновить вывод истории для трех режимов.

## Ошибки и безопасность

- Несуществующая или чужая генерация: сообщение `⚠️ Генерация не найдена.` без раскрытия чужих данных.
- Старая неполная запись: доступна для просмотра, но без кнопок сохранения/повтора из истории.
- Лимит шаблонов `10` сохраняется без изменения.
- Повтор и сохранение не используют output-текст как исходный запрос.
- Проверка владения записью выполняется при нажатии кнопки и повторно при финальном сохранении шаблона.

## Проверка

- Тест схемы БД: новые поля `generations` существуют.
- Тест сохранения: `save_successful_generation()` сохраняет метаданные и возвращает `generation_id`.
- Тест image-only: успешный image-run создает повторяемую запись истории.
- Тест клавиатур: новые результаты получают `generation_save:<id>` и `generation_repeat:<id>`.
- Тест ownership: загрузка для действий принимает только запись текущего пользователя.
- Тест сохранения: шаблон создается из input-полей записи, без output-полей.
- Тест повтора: адресная кнопка поднимает корректный input для всех трех режимов.
- Тест истории: выводит до пяти записей трех режимов и показывает кнопку только для строк с полным контекстом.
- Регрессия: legacy callbacks корректно обрабатывают старые сообщения.
- Полный запуск `pytest`.

## Вне scope

- Объединение технических отчетов `image_sessions` с пользовательским текстом истории.
- Сохранение уже существующих старых генераций как шаблонов через догадки о маркетплейсе.
- Изменение лимита шаблонов, тарификации, списания балансов или формата генерируемой карточки.
